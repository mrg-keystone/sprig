# Dev/prod parity — the two deploy incidents, and what sprig owes each

**Principle:** `sprig dev` must execute exactly what prod executes, so any prod defect shows
its ugly head locally first. Two alfred incidents violated this. Both are sprig; here is the
diagnosis, the owning layer, and the exact status.

This is the index doc. Incident 2's implementation is in `composition-parity.md`; incident
1's cache root cause + the maintainer's fix are in `bug-report.md` / `response.md`.

---

## Incident 1 — "inject() must be called synchronously" appeared in prod, never locally

**Symptom:** after a redeploy, a returning browser logged
`[sprig] failed to hydrate island "…" — inject() must be called synchronously…` for every
island. Fresh browsers and every local run were clean.

**Root cause (proven):** the browser reused a stale immutable-cached `client.js` from the
previous deploy next to freshly-fetched island chunks from the new deploy → **two copies of
the sprig client runtime** in one document. The active injector is module-scoped
(`framework/.sprig/core.ts:292 — let current`), so the copy the islands call has
`current === undefined` and `inject()` throws (`core.ts:297`). Local never showed it because
local caches were fresh — the textbook "prod-only, invisible locally" failure.

**Owner:** sprig, unambiguously. The error exists nowhere else (`grep` across keep → none);
rune's request/DI context uses `AsyncLocalStorage`, which is immune to both `await` and
dual-copy loss. Nothing about this is rune.

**Status: FIXED in 0.14.0.** Three layers, all sprig:
- Content-addressed `?v=` (the hash of the served `assetsDir`), so a redeploy changes asset
  URLs and old/new bundles can never coexist — the wedge's precondition is gone.
- `detectDualRuntime()` (`core.ts:272`) — if a second runtime copy ever loads, it logs a
  legible cause ("two copies… stale cached bundle after a redeploy") instead of the cryptic
  inject error.
- A once-per-session, `sessionStorage`-guarded recovery reload (`compiler/hydrate.ts`) that
  fetches a fresh document (new content-addressed URLs) instead of leaving islands dead.

**Parity note:** a *genuine* async-ordering bug — app code calling `inject()` after an
`await` — still throws, but it throws identically in dev and prod, so it surfaces locally as
required. The only prod-*only* variant was the dual-runtime wedge, now impossible + detected +
self-recovering. Nothing further needed. (Optional hardening: move `current` onto `globalThis`
like `__sprig_root` already is, closing the module-scope gap entirely.)

## Incident 2 — Google login worked locally, 404'd in prod

**Symptom:** the sign-in island called `/auth/firebase-config`; it worked against the local
dev server and returned 404 in prod ("auth not configured").

**Root cause (proven):** prod runs the **git-root `serve.ts`**, regenerated by
`sprig build --rune` on every deploy, which did not mount `/auth`. Local ran a *different*
entry (`ui/serve.ts`) that did. Two composition entrypoints, silently diverged — and worse,
`sprig dev` never served the prod composition at all: it built its handler from
`sprigUi({ app })` (UI only), so `/api`, `/docs`, and any composition-root route were absent
in dev but present in prod. The keep backend was identical in both — this is purely sprig's
serving/composition layer.

**Owner:** sprig.

**Status: FIXED on branch `feat/dev-prod-composition-parity`** (see `composition-parity.md`):
- `sprig dev` now composes the real `serveSprig({ keep, app })` when the UI is half of a rune
  monorepo — so `/api`, `/docs`, and any app-forwarded route are live in dev, HMR intact. If a
  route is missing, dev 404s it *locally*, first.
- The composition root becomes a **gitignored CLI artifact** — regenerated per deploy,
  composed in-process by dev, never committed — so there is one source of truth and no
  committed twin to drift.

**Remaining (small):** the divergence was possible because there were multiple places to mount
a route and only the `sprigApp` export is served by every host. Two follow-ups close the class:
1. A first-class extension seam on `ServeSprigConfig` (`before?: (req) => Response | null` or a
   routes map) so app routes like `/auth` don't need the "wrap `sprigApp`" trick.
2. Don't scaffold / relabel any secondary `*serve.ts` host entry — exactly one composition
   entry should exist.

## Answer to "rune, sprig, or both?"
**Both incidents: sprig.** Incident 1 is done (0.14.0). Incident 2 is done on the branch above
(needs a release + consumer bump). **Rune: nothing** — its DI is structurally immune to the
async-inject failure, and it is not the serving layer. The only rune-axis parity risk
(`Deno.openKv()` durable-local / ephemeral-prod) is now handled in infra (KV always
provisioned); optional rune boot-preflight is noted in `rune/feedback/alfred/dev-prod-parity.md`.
